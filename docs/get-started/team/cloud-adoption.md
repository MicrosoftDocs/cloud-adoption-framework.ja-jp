# <a name="get-started-build-a-cloud-adoption-team"></a>作業を開始しましょう。クラウド導入チームを構築する

クラウド導入チームは、技術実装チームやプロジェクト チームの現代版です。 ただし、クラウドの性質上、より流動的なチーム構成が必要になる場合があります。 クラウドの移行に集中的に取り組んでいるクラウド導入チームもあれば、クラウド テクノロジを利用したイノベーションに専念しているクラウド導入チームもあります。 また、データセンターの完全な移行など、大規模な導入作業を完了するために必要な幅広い技術に関する専門知識を有するチームもあれば、より専門的な技術に特化し、特定の目標を達成するためにプロジェクト間を移動するチームもあります。たとえば、SQL VM を SQL PaaS インスタンスに変換するのを支援するデータ プラットフォーム スペシャリストのチームなどです。

![クラウド導入チームの構築を開始する](../../_images/get-started/adoption-team-map.png)

## <a name="step-1-determine-the-type-of-adoption-team-you-need"></a>手順 1:必要な導入チームの種類を決定する

クラウド導入チームは、次の種類の導入を 1 件以上実施する傾向があります。

    - 既存のワークロードの移行。
    - 既存のワークロードと資産の最新化。
    - 既存のワークロードと資産のアーキテクチャの変更
    - 新しいワークロードの開発。

IT ポートフォリオの導入には、多くの場合、これらのスキルを組み合わせる必要があります。 残念ながら、これらの異なる作業にはそれぞれ異なるスキルとマインドセットが必要です。 これらの各種作業向けにより専門的な導入チームがあるようなら、そのチームがその種類の作業をより効果的かつ効率的に遂行できます。 逆に、クラウド導入全体のすべての実装オプションに習熟するのは圧倒されるような作業量になる可能性があります。

クラウド導入チームを最初に構築するときは、いずれかの導入手法に合わせることで、チームの集合的なスキルの開発を促進することができます。

**成果物:**

- チームが移行手法とイノベーション手法のどちらにより適しているかを判定します。
- 各手法には、その作業をうまくこなすために必要なツールとプロセスの把握に役立つ、4 ステップのオンボーディング エクスペリエンスがあります。 チームとして時間を確保して最初の数ステップを実行し、初期の繰り返しで最も必要になる可能性が高いツールとシナリオを把握します。
- 会社の [RACI テンプレート](../../organize/raci-alignment.md)を更新して、チームにだれが属しているか、そしてチームでどの手法の遂行に注力するかを他のメンバーが理解できるようにします。

**成果物の完遂をサポートするうえでのガイダンス:**

- [移行手法の概要](../../migrate/index.md)に関するページでは、ワークロードのポートフォリオを移行および最新化するためのプロセス、ツール、およびアプローチについて説明します。
- [イノベーション手法の概要](../../innovate/index.md)に関するページでは、クラウドネイティブのワークロードをポートフォリオに追加するためのプロセス、ツール、およびアプローチについて説明します。
- この取り組みの背後にある[動機を把握](../../strategy/motivations.md)して、移行やイノベーションの取り組みとより良く整合しているかどうかを確認します。

## <a name="step-2-align-your-team-to-other-supporting-teams"></a>手順 2:自分のチームと他のサポート チームを連携させる

会社のクラウド導入の取り組みがサポート チームを用意できるほど十分に成熟しているようであれば、一連のチームと各分野の専門家を自社バージョンの [RACI テンプレート](https://archcenter.blob.core.windows.net/cdn/fusion/management/raci-template.xlsx)で見つけられるかもしれません。たとえば、クラウド ガバナンス、クラウド運用、クラウドのセンター オブ エクセレンス、その他の類似のチームなどです。

**成果物:**

- さまざまなサポート チームの設計ガイダンス、運用ベースライン、ポリシー、およびプロセスを確認して、クラウド導入を正しく導くために確立されたガードレールを把握します。
- 他のクラウド導入チームとガイダンスを確認して、それらのガードレールの結果として発生する可能性がある制限事項を把握します。

**成果物の完遂をサポートするうえでのガイダンス:**

- 「[企業ポリシーの評価](../../govern/corporate-policy.md)」では、企業のポリシーを定義する手順の概要を説明しています。こうしたポリシーによって、会社のクラウド環境で安全に実行できる意思決定が制限される可能性があります。
- [ガバナンスの規範](../../govern/corporate-policy.md)に関するページでは、クラウドを安全かつコンプライアンスを守った方法で導入できるよう、ガバナンス チームが実装している可能性が高い制御や統制されたプロセスの種類について、概要を示しています。
- [管理手法](../../manage/index.md)に関するページでは、基本的な運用管理を提供するためにクラウド運用ベースラインに組み込まれる考慮事項について、概要を示しています。

## <a name="step-3-begin-your-adoption-journey"></a>手順 3:導入の取り組みを開始する

自分がメンバーとして属している導入チームの種類に応じて、次の 2 つの取り組みのどちらかかを開始します。

- 作業を開始しましょう。ワークロードをクラウドに移行する。
- 作業を開始しましょう。新しい製品またはサービスを構築する。

これらの各ファースト ステップ ガイドには、さまざまなチームのガイダンスが、さまざまなレベルの説明責任および責任と共に一覧表示されています。 これらをリファレンスとして使用して、自分のチームが残りの取り組みにどのように適合するかを把握してください。 また、これらのリファレンスを使用して、会社全体から得られるサポートのレベルを把握してください。

最終的に、クラウド導入チームは、割り当てられた移行作業や新製品開発の遂行について責任を負います。 サポート チームは各手順を確実に完了させる責任を負いますが、サポート チームが成功を収めるうえで必要なサポートを得られるようにするのは、各クラウド導入チームの責任です。 説明責任チームが存在しない場合や、責任を持つ手順を遂行するためにさらにサポートが必要な場合は、導入チームが他のチームと連携してこれらの手順を完了することをお勧めします。

**成果物:**

- 導入アプローチに関連する手法の遂行がますますスムーズに進むようになります。
- 他のチームが責任のある手順を完了するのをサポートします (それらの手順が導入の阻害要因になっている場合)。

**成果物の完遂をサポートするうえでのガイダンス:**

- 移行のファースト ステップ ガイドでは、導入チームが[手順 10: 最初のワークロードを移行する](../migrate.md#step-8-migrate-your-first-10-workloads)作業の遂行について責任を負います。
- 新製品のファースト ステップ ガイドでは、導入チームが[手順 8: クラウドでイノベーションを進める](../innovate.md#step-8-innovate-in-the-cloud)作業の遂行について責任を負います。

これらのチェックリストのその他すべての手順は、作業が管理しやすくなるように設計されています。

## <a name="step-4-expand-your-skills-with-scenarios-and-best-practices"></a>手順 4:シナリオとベスト プラクティスを使用してスキルを伸ばす

導入を 1、2 回繰り返した後、チームは主要な導入手法の基本を理解することとなります。 その後、チームはおそらく、追加のシナリオに取り組む準備が整い、いくつかの追加のベスト プラクティスの実装を開始できる状態になります。 次の各リンクを使用すると、ランディング ページが表示されます。ランディング ページには、チームがそのスキルを確認、強化するための両タイプのガイダンスの一覧が含まれる目次のセクションが表示されます。

**成果物:**

- より複雑な導入シナリオに対処するためのスキルを伸ばし、経験を積みます。

**成果物の完遂をサポートするうえでのガイダンス:**

- [シナリオ](../../migrate/azure-best-practices/contoso-migration-overview.md)と[ベスト プラクティス](../../migrate/azure-best-practices/index.md)を通じて、新しい種類のワークロードを移行するか、より複雑な移行の課題を解決します。
- [シナリオ](../../innovate/kubernetes/index.md)と[ベスト プラクティス](../../innovate/best-practices/index.md)を通じて、新しいクラウドネイティブのソリューションを使用してイノベーションを進めるか、より複雑なイノベーションの課題を解決します。

**説明責任チーム:**

クラウド導入チームは、スキルを強化する責任を負います。

## <a name="step-5-build-a-cloud-adoption-factory"></a>手順 5:クラウド導入ファクトリを構築する

チームがさまざまな導入シナリオに慣れるにつれ、より迅速に作業できるようになります。 ガイダンスのこのセクションでは、導入に関する能力を次のレベルに引き上げます。 クラウド導入ファクトリ アプローチでは、導入作業の背後にあるプロセスについて考察します。 移行とイノベーションに関連する時間的負担のほとんどは、理解と透明性が欠けているために数多くの会議が行われることに起因します。 クラウド導入の取り組みのさまざまな段階でプロセスと相互作用を明確に定義することで、文化的および政治的な阻害要因が取り除かれます。

**成果物:**

- 遂行プロセスを改善して、高度に最適化された導入ファクトリを作成します。

**成果物の完遂をサポートするうえでのガイダンス:**

- [移行作業](../../migrate/migration-considerations/index.md)をサポートするためのプロセス ガイダンスについては、移行手法のプロセスの改善に関するセクションを参照してください。
- イノベーション手法のガイダンスでは、テクノロジの削減とより影響力のある製品開発につながる[イノベーション プロセス](../../innovate/considerations/index.md)に焦点を当てています。

**説明責任チーム:**

クラウド導入チームは、導入を次のレベルに引き上げるプロセスを構築する責任を負います。

## <a name="whats-next"></a>参照トピック

導入は多くの利点をもたらしますが、管理されていない導入によって予期しない結果が生じる場合があります。 クラウド導入を[クラウド ガバナンス機能](../../organize/cloud-governance.md)と連携させて、ビジネスとテクノロジのリスクを軽減しながら、導入とベスト プラクティスに関する取り組みを促進します。

これら 2 つのチームは、クラウド導入に伴う各取り組みのバランスを取るものの、持続可能でない可能性があるため、MVP と見なされます。 ["_実行責任、説明責任、助言、通知_" (RACI) チャート](../../organize/raci-alignment.md)で示されるように、各チームが多くの役割を担います。

「[組織のアンチパターン: サイロと封土](../../organize/fiefdoms-silos.md)」で詳細を確認してください。
