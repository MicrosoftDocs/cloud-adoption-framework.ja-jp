# <a name="empower-adoption"></a>導入の強化

イノベーションの最終的なテストは、その発明に対する顧客の反応です。 仮説は正しかったのか。 顧客はこのソリューションを使用するのか。 期待する割合のユーザーのニーズに合うようにスケーリングされているのか。 最も重要なこととして、ユーザーは使い続けてくれるのか。 これらの質問は、MVP ソリューションがデプロイされるまでは確認できません。 この記事では、導入の強化の規範に焦点を絞って説明します。

## <a name="reducing-friction-to-adoption"></a>導入に対する摩擦の削減

テクノロジとプロセスを組み合わせることによって最小限に抑えることができる、導入に対する重要な摩擦ポイントがいくつかあります。 CI/CD または DevOps プロセスに精通している読者は、以下の説明は非常に似ていると思われるでしょう。 この記事の焦点は、クラウド導入チームのために出発点を確立するということです。これにより、イノベーションとフィードバック ループが促進されます。 長期的には、この出発点は、製品やチームが成熟するにつれて、より堅牢な CI/CD または DevOps アプローチに発展する可能性があります。

「[顧客への影響の測定](./measure.md)」で説明されているとおり、仮説の有益な検証には、反復と、強い意志で臨むことが必要です。 イノベーションのサイクル中は、成功よりも失敗の数の方が多くなるでしょう。 これは予期されることです。 とは言え、顧客のニーズ、仮説、およびソリューションが大規模なものになると、世界は急速に変化します。 この記事では、イノベーションを失速させる可能性のある[技術的スパイク](./build.md#reduce-complexity-and-delay-technical-spikes)を最小限に抑えることを目的としていますが、いくつかの信頼性の高いベスト プラクティスも用意されています。 これにより、チームは将来の成功に向けてのデザインをする一方で、現在の顧客のニーズに対応することもできます。

## <a name="empowering-adoption---maturity-model"></a>導入の強化 - 成熟度モデル

[イノベーション方法論](./index.md)の主な目的は、顧客パートナーシップを構築し、フィードバック ループを加速させることです。これにより、市場のイノベーションが実現します。
次のイメージおよびコンテンツ セクションでは、方法論をサポートする初期の実装について説明します。

![導入の強化 - 成熟度モデル](../../_images/innovate/empower-adoption-maturity.png)

- [共有ソリューション](#shared-solution): ソリューションのあらゆる側面に対応する一元的なリポジトリを確立します。
- [フィードバック ループ](#feedback-loops): フィードバック ループが、反復を通じて一貫して確実に管理されるようにします。
- [継続的インテグレーション](#continuous-integration): ソリューションを定期的にビルドして統合します。
- [信頼性の高いテスト](#reliable-testing): ソリューションの品質と予想される変更を検証して、対策を確保します。
- [ソリューションのデプロイ](#solution-deployment): チームが変更を顧客に迅速に共有できるようにするための、ソリューションのデプロイ。
- [統合された測定](#integrated-measurements): 完全なチームによる明確な分析のために、学習メトリックをフィードバック ループに追加します。

技術的スパイクを最小限に抑えるため、これらの各原則において成熟度は最初は低いものであると仮定します。 ただし、仮説がより詳細になるにつれてスケーリングが可能なツールとプロセスを調整して、事前に計画します。 Azure では、[GitHub](https://guides.github.com) と [Azure DevOps](https://docs.microsoft.com/azure/devops) を組み合わせることで、小規模なチームで摩擦をほぼ生じさせずに作業を開始することができます。 その後、何千もの開発者が、何百もの顧客の仮説をテストするスケーリング ソリューションで共同作業を行う規模に発展していきます。 この記事の残りの部分では、これらの各原則を通じて導入を強化するための、大きく計画して小さく開始するというアプローチについて説明します。

## <a name="shared-solution"></a>共有ソリューション

「[顧客への影響の測定](./measure.md)」で説明されているとおり、仮説の有益な検証には、反復と、強い意志で臨むことが必要です。 イノベーションのサイクル中は、成功よりも失敗の数の方が多くなるでしょう。 これは予期されることです。 とは言え、顧客のニーズ、仮説、およびソリューションが大規模なものになると、世界は急速に変化します。

イノベーションをスケーリングする場合、ソリューションの共有コードベースよりも有益なツールはありません。 残念ながら、どの反復または MVP の組み合わせが成功を実現するのを予測する方法はありません。 このため、共有コードベースまたはリポジトリはいつ確立しても早すぎるということはありません。 これは、遅滞すべきでない[技術的スパイク](./build.md#reduce-complexity-and-delay-technical-spikes)の 1 つです。 チームはさまざまな MVP ソリューションを反復処理するため、共有リポジトリを使用すると、共同作業や迅速な開発を行うことができます。 ソリューションへの変更によって学習メトリックに悪影響が及ぶ場合は、バージョン コントロールを使用して、以前のより効果的なバージョンのソリューションにロールバックすることができます。

コード リポジトリを管理するための最も一般的なツールは [GitHub](https://guides.github.com) です。これにより、数回のクリックで共有コード リポジトリを作成することができます。 また、Azure DevOps の [Azure Repos](https://docs.microsoft.com/azure/devops/repos/get-started/what-is-repos?view=azure-devops) 機能を使用して、[Git](https://docs.microsoft.com/azure/devops/repos/get-started/what-is-repos?view=azure-devops#git) または [Team Foundation](https://docs.microsoft.com/azure/devops/repos/get-started/what-is-repos?view=azure-devops#tfvc) リポジトリを作成することもできます。

## <a name="feedback-loops"></a>フィードバック ループ

イノベーション サイクル中に顧客パートナーシップを構築する鍵は、顧客をソリューションの一部にすることです。 これは、[顧客への影響を測定](./measure.md)することによって、ある程度行われます。 さらに深めるには、顧客と会話し直接テストを行うことが必要です。 どちらも、管理が必要となるフィードバックを得ることができます。

すべてのフィードバック ポイントは、顧客のニーズに対する解決策となる可能性を秘めています。 さらに重要なこととして、顧客から直接フィードバックをもらうことはすべて、パートナーシップを改善する機会となります。 フィードバックが MVP ソリューションに取り入れられたら、顧客と喜びを分かち合いましょう。 フィードバックに対応できない場合であっても、そのフィードバックを取り上げないという決定を開示し、[成長の思考様式](./learn.md#growth-mindset)と[継続的な学習](./learn.md#continuous-learning)への取り組みを示します。

[Azure DevOps](https://docs.microsoft.com/azure/devops) には、[フィードバックの要求、提供、管理を行う](https://docs.microsoft.com/azure/devops/project/feedback)ための方法が含まれています。 これらの各ツールによりフィードバックを一か所にまとめることができるため、チームはフィードバックに対してアクションの実行や報告を行うことができます。これにより、透明性のあるフィードバック ループが実現します。

## <a name="continuous-integration"></a>継続的インテグレーション

導入の規模と仮説が大規模な真のイノベーションに近づくにつれて、テスト対象となる小さな仮説の数が急速に増加します。 正確なフィードバック ループとスムーズな導入プロセスを実現するには、これらの各仮説が統合され、イノベーションの背後にある主要な仮説を支えることが重要です。 残念ながら、イノベーションと成長のためにはスピードも重要となります。そのためには、複数の開発者が核となる仮説のバリエーションをテストする必要があります。 開発作業の後半の段階では、1 つの共有ソリューションに向けて構築を行う複数の開発者チームが必要になることがあります。 継続的インテグレーションは、多岐にわたる動きを管理するための最初の一歩となります。

継続的インテグレーションでは、コードの変更がメイン ブランチに頻繁にマージされます。 自動化されたビルドとテストのプロセスによって、確実にメイン ブランチのコードが、常に運用環境の品質であるようにします。 これにより、開発者は共同作業を行って、正確で信頼性の高いフィードバック ループを提供する共有ソリューションを開発することができます。

Azure DevOps と [Azure Pipelines](https://docs.microsoft.com/azure/devops/pipelines) は、GitHub またはその他のさまざまなリポジトリに対して、数回のクリックで継続インテグレーション機能を提供します。
[継続的インテグレーション](https://docs.microsoft.com/azure/devops/learn/what-is-continuous-integration)の詳細について確認してください。または、[ハンズオン ラボ](https://www.azuredevopslabs.com/labs/azuredevops/continuousintegration)で詳細を参照してください。 [Azure DevOps を使用した CI/CD パイプライン](https://azure.microsoft.com/solutions/devops)の作成を高速化する、ソリューション アーキテクチャも提供されています。

## <a name="reliable-testing"></a>信頼性の高いテスト

どのソリューションでも欠陥によって、擬陽性や偽陰性が発生する可能性があります。 予期しないエラーが発生すると、ユーザー導入のメトリックが誤って解釈される可能性があります。 また、仮説のテストとはずれている顧客からの否定的なフィードバックにつながる可能性もあります。

MVP ソリューションの初期の反復においては欠陥は想定されており、初期に導入を行う人にとっては歓迎すべきものである場合もあります。 初期リリースにおいては、承認テストが存在しない可能性があります。 しかしながら、ニーズと仮説の検証は、共感を得ながら構築することの 1 つの側面です。 どちらも、コード レベルの単体テストと、デプロイ前の手動承認テストを介して完了可能です。 これらを組み合わせることで、テストの信頼性を高めることができます。 長期的には、明確に定義された一連のビルド、単体、および承認テストを自動化して、仮説および結果であるソリューションに対するより詳細な調整に関連する信頼性の高いメトリックを確保する必要があります。

[Azure Test Plans](https://docs.microsoft.com/azure/devops/test/track-test-status?view=azure-devops) には、手動または自動のテスト実行中にテスト計画を開発および操作するためのツールが用意されています。

## <a name="solution-deployment"></a>ソリューションのデプロイ

導入を強化する最も意味のある要素はおそらく、ソリューションの顧客へのリリースを制御する機能です。 ソリューションを顧客にリリースするためのセルフサービスまたは自動化されたパイプラインの提供により、フィードバック ループが加速します。 顧客がソリューションの変更をすぐに操作できるようにすることで、顧客をプロセスに取り込むことができます。 また、仮説の迅速なテストが可能になり、前提条件と想定される再作業を減らすことができます。

ソリューションのデプロイにはいくつかのアプローチがありますが、最も一般的な 3 つのアプローチの概要を次に示します。

- 最も高度なアプローチである**継続的デプロイ**により、コード変更を運用環境に自動的にデプロイできます。 成熟したチームが成熟した仮説をテストするためには、継続的デプロイは非常に有効です。
- 開発の初期段階では、**継続的デリバリー**の方が適している場合があります。 継続的デリバリーでは、コード変更は運用環境に似た環境に自動的にデプロイされます。 開発者、ビジネスの意思決定者、およびチームの他のユーザーは、この環境を使用して、作業内容が実稼働可能かどうかを検証できます。 また、現行のビジネス アクティビティに影響を与えずに、顧客と仮説をテストするために使用することもできます。
- **手動デプロイ**は、リリース管理のための最も洗練されていないアプローチです。 名前が示すように、チームのだれかが、最新のコード変更を手動でデプロイします。 このアプローチはエラーが発生しやすく、信頼性が低いものです。また、ほとんどの経験豊富なエンジニアはアンチパターンであると考えます。

上記の説明の一方で、手動デプロイは MVP ソリューションの最初の反復においては一般的なアプローチです。 ソリューションが非常に流動的で、顧客からのフィードバックが不明な場合は、ソリューション全体 (または核となる仮説) をリセットすることには重大なリスクがあります。 手動デプロイの一般的なルールは次のとおりです。顧客によるテストは実施しない、デプロイの自動化は行わない。 早期の投資は、時間損失につながる可能性があります。 さらに重要なことは、リリース パイプラインに依存関係を作成すると、チームが初期の方向転換に対して耐性を持てるようになることです。 最初の数回の反復を終えた後、または顧客からのフィードバックが成功の可能性を示しているときは、より高度なデプロイ モデルを早急に導入する必要があります。

仮説検証のどの段階でも、Azure DevOps と [Azure Pipelines](https://docs.microsoft.com/azure/devops/pipelines) は、わずか数回のクリックで継続的デリバリーと継続的デプロイ機能を提供します。 [継続的デリバリー](https://docs.microsoft.com/azure/devops/learn/what-is-continuous-delivery)の詳細について確認してください。または、[ハンズオン ラボ](https://www.azuredevopslabs.com/labs/azuredevops/continuousdeployment)で詳細を参照してください。 ソリューション アーキテクチャにより、[Azure DevOps を使用した CI/CD パイプライン](https://azure.microsoft.com/solutions/devops)の作成を高速化することもできます。

## <a name="integrated-measurements"></a>統合された測定

[顧客への影響を測定](./measure.md)場合は、ソリューションの変更に対する顧客の反応を理解することが重要です。 利用統計情報と呼ばれるこのデータでは、ユーザー (またはユーザーのコーホート) がソリューションの使用時に行ったアクションに関する分析情報が提供されます。 このデータから、仮説の定量的な検証を簡単に取得できます。 これらのメトリックを使用して、ソリューションを調整し、より詳細な仮説を生成できます。 これらの微妙な変更により、初期ソリューションがその後の反復でさらに成熟し、最終的に大規模な導入を繰り返すことができます。

Azure では、[Azure Monitor](https://docs.microsoft.com/azure/azure-monitor/overview) に、カスタマー エクスペリエンスからデータを収集して確認するためのツールとインターフェイスが用意されています。 これらの観測と分析情報を使用して、[Azure Boards](https://docs.microsoft.com/azure/devops/boards) でバックログを調整することができます。
